25. 지원 번들 (Support Bundle)

에어갭(Air-gapped) 환경에서 원격 진단이 불가능할 때, 시스템 상태를 한 번에 수집하여 tar.gz 파일로 내보내는 기능입니다. 번들 생성 시 PII(개인식별정보) 산화가 자동 적용되어 민감정보가 안전하게 처리됩니다.

25.1 개요​

사이드바에서 도구 > 지원 번들을 선택합니다. 시스템 진단 정보를 수집하여 하나의 압축 파일로 다운로드할 수 있습니다.

활용 시나리오
  • 기술 지원 요청 시 시스템 상태 첨부
  • 장애 발생 시 로그/설정/메트릭 일괄 수집
  • 정기 점검 시 시스템 스냅샷 보관

25.2 번들 생성​

번들 생성 버튼을 클릭하면 확인 다이얼로그가 나타납니다. 확인 후 수집이 시작되며 진행 상태가 실시간으로 표시됩니다.

수집 카테고리​

카테고리수집 내용저장 위치
시스템 정보호스트명, OS, 커널, 아키텍처, 업타임, 타임존system/info.json
버전 정보tkadmin, 엔진(MariaDB/Redis/Nginx/OpenSearch), TACHYON 버전system/versions.json
서비스 상태서비스별 Active 상태, PID, CPU/메모리 사용량system/services.json
시스템 로그tkadmin 로그 파일 (.log, .log.gz)logs/tkadmin/
서비스 로그주요 서비스 로그 파일 + journalctl 출력logs/services/
설정 파일tkadmin, TACHYON, nginx, MariaDB, Redis, OpenSearch 설정configs/{category}/
감사 로그최근 30일 감사 로그 (JSONL 형식, 최대 10만건)logs/tkadmin/audit.jsonl
리소스 메트릭CPU/메모리/디스크 현황 + 용량 스냅샷(90일)system/metrics.json, capacity/snapshots.jsonl

진행 상태 표시​

수집 진행 중 각 카테고리별 상태가 아이콘으로 표시됩니다:

  • 대기: 아직 수집 시작 전
  • 수집 중: 현재 데이터 수집 진행 중 (스피너)
  • 완료: 수집 성공
  • 실패: 수집 중 오류 발생

25.3 수집 기간 설정​

번들 생성 시 수집할 로그의 기간을 설정할 수 있습니다. 기간을 좁히면 번들 크기가 작아지고 생성 시간이 단축됩니다.

프리셋 기간​

번들 생성 화면에서 다음 프리셋 중 하나를 선택합니다:

프리셋수집 범위권장 상황
최근 1일24시간 이내 로그방금 발생한 장애 분석
최근 7일7일 이내 로그 (기본값)일반적인 기술지원 요청
최근 30일30일 이내 로그간헐적 장애, 추이 분석

사용자 정의 기간​

프리셋이 적합하지 않은 경우 사용자 정의를 선택하여 시작일과 종료일을 직접 지정할 수 있습니다.

  1. 사용자 정의 라디오 버튼을 선택합니다.
  2. 시작일과 종료일 날짜 선택기에서 원하는 범위를 지정합니다.
  3. 기간을 변경하면 예상 크기가 자동으로 갱신됩니다.

예상 크기 미리보기​

수집 기간이나 카테고리를 변경할 때마다 예상 번들 크기가 자동으로 계산되어 프로그레스 바로 표시됩니다.

  • 카테고리별 예상 크기를 확인하여 불필요한 항목을 제외할 수 있습니다.
  • 예상 크기는 참고용이며 실제 크기와 다를 수 있습니다.
번들 크기 줄이기

기간을 1일로 설정하고 필요한 카테고리만 선택하면 번들 크기를 최소화할 수 있습니다. 특히 서비스 로그가 대부분의 용량을 차지하므로, 문제가 특정 서비스에 한정된 경우 해당 카테고리만 선택하는 것을 권장합니다.

로그 로테이션 파일 포함​

수집 기간 내에 해당하는 로그 로테이션 파일(.gz 압축 파일, log4j 날짜별 로그 등)도 자동으로 포함됩니다.

  • logrotate 압축 파일 (.log.1.gz, .log.2.gz): 기간 범위에 해당하면 자동 포함
  • log4j 날짜 기반 로그 (Api1.2026-03-10.log): 날짜 패턴으로 필터링
  • journalctl 로그: --since 옵션으로 기간 제한
압축 파일의 PII 산화

금융 또는 군/국방 산화 프로파일을 사용하는 경우, 압축 로그 파일도 자동으로 PII 산화가 적용됩니다. 6종 압축 포맷을 지원하며, 모든 라이브러리는 순수 Go 구현으로 바이너리에 내장되어 에어갭 환경에서도 동작합니다.

포맷처리 방식비고
.gz (gzip)스트리밍 해제 → 라인 단위 산화 → gzip 재압축원본 포맷 유지
.zip아카이브 내 텍스트 파일(.log, .txt, .yml, .json, .xml, .properties, .conf)만 산화, 바이너리 엔트리는 그대로 유지원본 포맷 유지
.bz2 (bzip2)해제 → 산화 → .gz로 포맷 변환 (.bz2.redacted.gz 파일 생성)Go stdlib에 bzip2 Writer 미제공
.xz스트리밍 해제 → 라인 단위 산화 → xz 재압축원본 포맷 유지
.zst/.zstd스트리밍 해제 → 라인 단위 산화 → zstd 재압축원본 포맷 유지
.7z해제 → 산화 → .zip으로 포맷 변환 (.7z.redacted.zip 파일 생성)순수 Go 라이브러리가 Read only

포맷 변환 안내: .bz2와 .7z는 사용하는 순수 Go 라이브러리가 읽기만 지원하므로, 산화 후 각각 .gz, .zip으로 변환됩니다. 변환된 파일명에 원본 확장자가 포함되어 있어 원래 포맷을 식별할 수 있습니다.

압축 파일 산화가 활성화되면 번들 생성 시간이 다소 늘어날 수 있습니다.

기타 프로파일(표준, 공공기관, 의료, GDPR)에서는 압축 파일이 원본 그대로 포함되며, manifest.json에 해당 파일 목록과 함께 compressed_files_redacted: false 경고가 표기됩니다. 필요한 경우 프로파일 설정에서 압축 파일 산화 옵션을 수동으로 활성화할 수 있습니다.

25.4 번들 이력​

생성된 번들은 이력 테이블에 표시됩니다. 각 항목에는 다음 정보가 포함됩니다:

컬럼설명
ID번들 고유 식별자 (예: sb-20260312-143500)
생성 일시번들 생성 시각
상태ready (다운로드 가능), generating (생성 중), error (실패)
크기압축 파일 크기
PII 산화산화 처리된 항목 건수 배지
다운로드다운로드 버튼 (ready 상태에서만 활성)

저장소 통계​

이력 카드 헤더에 저장소 통계 바가 표시됩니다. 현재 저장소 사용 현황을 한눈에 확인할 수 있습니다:

항목설명
총 크기모든 번들의 합산 디스크 사용량
번들 수현재 저장된 번들 개수 / 최대 보관 수
최오래된 번들가장 오래된 번들의 경과 시간
정리 버튼만료된 번들을 즉시 수동 정리 (권한 상승 필요)

자동 정리 정책 (Lifecycle Policy)​

번들 저장소는 3중 생명주기 정책으로 자동 관리됩니다. 6시간 주기로 스케줄러가 실행되어 만료 조건에 해당하는 번들을 자동 삭제합니다.

정책기본값설명
최대 번들 수 (MaxBundles)10초과 시 오래된 번들부터 삭제
최대 보관 기간 (MaxAge)30일기간 초과 번들 삭제
디스크 사용률 제한 (MaxDiskUsage)80%임계치 초과 시 오래된 번들부터 삭제
최소 유지 수 (MinRetain)1정리 시에도 최소 1개는 보존
수동 정리

이력 카드 헤더의 정리 버튼을 클릭하면 자동 정리 스케줄러와 동일한 정책으로 즉시 정리를 수행합니다. 정리 결과(삭제 건수, 확보된 공간)가 토스트 알림으로 표시됩니다.

25.5 번들 다운로드​

이력 테이블에서 다운로드 버튼을 클릭하면 support-bundle-{hostname}-{timestamp}.tar.gz 파일이 다운로드됩니다.

번들 내부 구조​

support-bundle-tums4-20260312-143500.tar.gz
├── manifest.json          # 메타데이터 (호스트명, 생성시각, 카테고리 목록)
├── system/
│   ├── info.json          # 시스템 정보
│   ├── versions.json      # 버전 정보
│   ├── services.json      # 서비스 상태
│   └── metrics.json       # 리소스 메트릭
├── logs/
│   ├── tkadmin/
│   │   ├── tkadmin.log    # 시스템 로그
│   │   └── audit.jsonl    # 감사 로그
│   └── services/
│       ├── mariadb.log    # 서비스별 로그
│       └── ...
├── configs/
│   ├── tkadmin/           # tkadmin 설정
│   ├── tachyon/           # TACHYON 설정 (YML)
│   ├── nginx/             # nginx 설정
│   └── ...
├── redaction-report.json  # PII 산화 처리 결과 리포트
└── capacity/
    └── snapshots.jsonl    # 용량 스냅샷 이력

25.6 보안 고려사항​

민감 정보 보호

번들 생성 시 다음과 같은 보안 조치가 자동 적용됩니다:

  • 파일 제외: 인증서/키 파일 (.key, .crt, .pem, .p12)은 수집에서 제외
  • PII 산화: 로그·설정 파일 내 비밀번호, 개인정보, IP 주소 등이 자동으로 마스킹 처리 (자세한 내용은 25.7 PII 민감정보 산화 참조)
  • 암호화값 보존: [enc] ARIA 암호화값은 이미 보호 상태이므로 원본 유지

번들 파일에는 시스템 설정과 로그가 포함되므로, 전달 시 보안에 유의하시기 바랍니다.

25.7 PII 민감정보 산화​

번들 생성 시 로그·설정 파일에 포함된 개인식별정보(PII)가 자동으로 산화(redaction)됩니다. 산화란 민감정보를 마스킹·치환하여 원본 값을 복원할 수 없도록 처리하는 과정입니다.

산화 레벨​

레벨분류대상산화 방식선택
L1인증정보password, api_key, token, DSN완전 치환 (***REDACTED***)필수
L2한국 PII주민등록번호, 사업자번호, 휴대폰번호부분 마스킹 (010-****-5678)필수
L3네트워크IPv4, MAC 주소IP: 옥텟 마스킹/가명화, MAC: 마스킹선택
L4사용자 식별이메일 주소부분 마스킹선택
L3/L4 선택 옵션

번들 생성 화면의 PII 필터링 옵션 패널에서 L3(네트워크)과 L4(사용자 식별) 산화를 개별적으로 활성화/비활성화할 수 있습니다. L1과 L2는 항상 적용됩니다.

프로파일 상세 보기​

각 산화 프로파일 카드에 표시된 ? 아이콘 버튼을 클릭하면 해당 프로파일에 적용되는 산화 패턴의 상세 정보를 확인할 수 있습니다.

상세 팝업 테이블:

컬럼설명
대상산화 대상 항목 (비밀번호, 주민등록번호, IP 등)
기법ISO 20889 분류에 따른 산화 기법
산화 예시산화 전→후 변환 예시
국제 표준해당 기법의 국제 표준 근거

기법은 색상으로 구분됩니다:

  • 삭제/억제 (빨강) — 값을 완전히 제거하거나 고정값으로 치환
  • 부분 마스킹 (황색) — 일부 문자만 마스킹하여 형식 유지
  • 일관 가명화 (파랑) — 동일 원본을 동일 가명으로 일관 치환
압축산화 표시

금융 또는 군/국방 프로파일에는 빨간색 압축산화 배지가 표시됩니다. 이 프로파일이 선택되면 압축 파일 내부까지 PII 산화가 적용되며, 번들 생성 시간이 다소 늘어날 수 있다는 안내가 함께 표시됩니다.

산화 전/후 예시​

구분산화 전산화 후
비밀번호password=MySecret123password=***REDACTED***
휴대폰번호010-1234-5678010-****-5678
주민등록번호900101-1234567900101-*******
사설 IP192.168.139.100192.168.139.*
공인 IP203.0.113.50[IP-001]
이메일 (L4)user@example.comu***@example.com
IP 산화 정책
  • 사설 IP (192.168.x.x, 10.x.x.x, 172.16~31.x.x): 마지막 옥텟만 마스킹
  • 공인 IP: 일관된 가명화 처리 (동일 IP는 동일 코드로 치환)
  • 특수 IP (0.0.0.0, 127.0.0.1 등): 산화하지 않음

25.8 산화 결과 확인​

번들 생성이 완료되면 산화 결과 패널이 표시됩니다.

결과 패널 구성​

  • 레벨별 배지: L1~L4 각 레벨의 산화 처리 건수
  • 패턴별 건수: 비밀번호, 주민번호, IP 주소 등 패턴별 상세 건수
  • 검증 배지: 산화 후 재스캔을 수행하여 L1+L2 항목이 잔존하지 않음을 확인한 결과 (PASS/FAIL)

이력 테이블에서 상세 보기​

이력 테이블에서 번들 행을 클릭하면 아코디언 패널이 확장되어 다음 정보를 확인할 수 있습니다:

  1. 요약 카드 — 번들 메타데이터
  2. 카테고리 태그 — 수집된 데이터 카테고리
  3. PII 산화 결과 — 레벨별 건수 및 검증 결과
  4. DIFF 비교 뷰 — 산화 전/후 텍스트 비교

25.9 산화 제외 항목​

다음 항목은 산화 대상에서 자동으로 제외됩니다:

제외 항목설명
[enc] 암호화값ARIA 암호화가 적용된 값은 이미 보호 상태이므로 산화하지 않음
null/빈 비밀번호password= 또는 password=null처럼 값이 비어있는 경우
특수 IP0.0.0.0, 127.0.0.1 등 RFC 5735 정의 특수 목적 주소
암호화값 보존

[enc]로 시작하는 TACHYON 암호화 값은 산화하지 않습니다. 이 값은 ARIA 알고리즘으로 암호화되어 있어 복호화 키 없이는 원본을 알 수 없으므로, 산화 처리가 불필요합니다.

25.10 redaction-report.json​

각 번들에는 산화 처리 결과를 기록한 redaction-report.json 파일이 포함됩니다.

리포트 주요 항목​

항목설명
패턴별 집계각 산화 패턴(password, phone, ip 등)별 처리 건수
DIFF 비교산화 전/후 변경 내역 (파일별)
오탐 후보산화 패턴과 일치하지만 실제 PII가 아닐 수 있는 항목
검증 결과산화 후 재스캔 결과 (L1+L2 잔존 건수 확인)
리포트 확인 방법

이력 테이블에서 번들을 클릭하여 확장 패널의 DIFF 비교 뷰에서 시각적으로 확인하거나, 다운로드한 번들 내 redaction-report.json 파일을 직접 열어볼 수 있습니다.
