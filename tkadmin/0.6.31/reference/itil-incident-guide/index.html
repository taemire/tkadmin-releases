ITIL v4 기반 인시던트 대응 가이드

tkadmin 알림 보고의 조치 권고 사항은 ITIL v4, NIST SP 800-61 Rev.3, ISO/IEC 27035, ISO/IEC 20000-1:2018을 준거 표준으로 설계되었습니다.

1. ITIL v4 우선순위 매트릭스​

ITIL v4(Information Technology Infrastructure Library v4)는 IT 서비스 관리를 위한 국제 모범 사례 프레임워크입니다. 인시던트 우선순위는 **영향도(Impact) × 긴급도(Urgency)**의 매트릭스로 결정됩니다.

1.1 우선순위 정의​

우선순위이름최대 응답 시간최대 해결 시간tkadmin 이벤트
P1Critical15분4시간FAILURE, RECOVERY_FAILED
P2High30분8시간CONFIG_UPDATE
P3Medium2시간72시간RECOVERY_SUCCESS
P4Low8시간2주SERVICE_RESTART

1.2 영향도 × 긴급도 매트릭스​

                    긴급도
                 높음    중간    낮음
영향도  높음  │  P1  │  P2  │  P3  │
        중간  │  P2  │  P3  │  P4  │
        낮음  │  P3  │  P4  │  P4  │

1.3 tkadmin 이벤트별 우선순위 근거​

FAILURE → P1 Critical​

  • 영향도: 높음 — 서비스가 완전 중단되어 사용자·업무에 즉각 영향
  • 긴급도: 높음 — 서비스 중단은 시간이 지날수록 피해 확대
  • SLA 기준: 15분 내 첫 응답, 4시간 내 서비스 복구

RECOVERY_FAILED → P1 Critical​

  • 영향도: 높음 — 자동 복구도 실패하여 서비스 중단 상태 지속
  • 긴급도: 높음 — 추가 수동 개입 없이 자동 해결 불가
  • 특이사항: NIST SP 800-61 Rev.3에 따라 보안 침해 가능성도 검토 필요

CONFIG_UPDATE → P2 High​

  • 영향도: 중간 — 잘못된 설정이 서비스 불안정으로 이어질 수 있음
  • 긴급도: 높음 — 변경 직후 검토가 필요하나 즉각 장애는 아님
  • ITIL 프레임워크: Change Enablement — 모든 변경은 추적·승인·검토 필요

RECOVERY_SUCCESS → P3 Medium​

  • 영향도: 낮음 — 서비스가 복구되었으나 재발 가능성 존재
  • 긴급도: 중간 — 지속 모니터링 필요하나 즉각 대응 불필요
  • 후속 조치: MTTR 기록, PIR(사후 검토), 재발 방지 계획

SERVICE_RESTART → P4 Low​

  • 영향도: 낮음 — 계획된 재시작은 정상 운영 범위
  • 긴급도: 낮음 — 확인·기록 수준의 관리 필요
  • ITIL 프레임워크: Change Enablement — 표준 변경(Standard Change) 범주

2. MTTR (Mean Time To Recovery)​

2.1 MTTR이란?​

**MTTR (Mean Time To Recovery / Mean Time To Repair)**는 시스템이 장애 발생 후 정상 상태로 복구되는 데 걸리는 평균 시간입니다. ITIL v4와 ISO/IEC 20000-1:2018에서 서비스 수준 측정의 핵심 지표로 사용됩니다.

MTTR = 총 복구 소요 시간 / 장애 발생 건수

2.2 관련 지표​

지표전체 이름의미
MTTRMean Time To Recovery장애 감지 → 서비스 복구까지 평균 시간
MTBFMean Time Between Failures두 장애 사이의 평균 정상 운영 시간
MTTDMean Time To Detect장애 발생 → 감지까지 평균 시간
MTTAMean Time To Acknowledge감지 → 담당자 첫 응답까지 평균 시간
MTTFMean Time To Failure시스템이 첫 장애를 일으키기까지 평균 시간

2.3 tkadmin에서의 MTTR 측정​

tkadmin Watchdog은 FAILURE 감지 시각과 RECOVERY_SUCCESS 시각을 audit_logs에 기록합니다.

MTTR (단일 장애) = RECOVERY_SUCCESS.time - FAILURE.time

MTTR 기록 방법:

  1. RECOVERY_SUCCESS 알림 다이얼로그 열기
  2. 조치 권고 사항의 "MTTR 기록" 항목 확인
  3. 시스템 로그에서 FAILURE 시각 조회
  4. RECOVERY_SUCCESS 시각에서 FAILURE 시각을 빼서 MTTR 계산
  5. 인시던트 레코드에 기록

2.4 MTTR 개선 전략​

단계전략기대 효과
탐지 단축Watchdog 폴링 주기 최적화 (기본 30초)MTTD 감소
응답 단축P1 알림 즉시 에스컬레이션 채널 설정MTTA 감소
복구 단축자동 복구 스크립트 고도화 (watchdog.yml)자동 복구 MTTR 감소
재발 방지PIR(사후 검토) → RCA(근본 원인 분석)MTBF 증가
RECOVERY_SUCCESS 후 반드시 MTTR을 기록하세요

MTTR 트렌드 분석을 통해 서비스 안정성 개선 우선순위를 파악할 수 있습니다. 목표 MTTR을 SLA에 명시하고 주기적으로 검토하는 것을 권장합니다.

3. NIST SP 800-61 Rev.3 — 인시던트 대응 라이프사이클​

NIST SP 800-61 Rev.3(Computer Security Incident Handling Guide)은 미국 국립표준기술원(NIST)이 발간한 컴퓨터 보안 인시던트 대응 가이드입니다. tkadmin의 RECOVERY_FAILED 권고 사항은 이 표준의 대응 절차를 따릅니다.

3.1 인시던트 대응 4단계​

준비(Preparation)
    ↓
탐지·분석(Detection & Analysis)   ← Watchdog이 자동 수행
    ↓
억제·근절·복구(Containment, Eradication & Recovery)   ← 조치 권고 사항 기반
    ↓
사후 활동(Post-Incident Activity)   ← PIR, MTTR 기록

3.2 단계별 tkadmin 연계​

단계 1: 준비 (Preparation)​

  • Watchdog 서비스 감시 설정 (watchdog.yml)
  • 인시던트 대응 팀 연락처 사전 등록
  • 에스컬레이션 경로 정의

단계 2: 탐지·분석 (Detection & Analysis)​

  • Watchdog이 서비스 이상 탐지 → FAILURE 알림 생성
  • 자동 복구 시도 → RECOVERY_SUCCESS 또는 RECOVERY_FAILED
  • tkadmin 조치: 알림 보고에서 이벤트 상세 확인, 로그 분석

단계 3: 억제·근절·복구 (Containment, Eradication & Recovery)​

RECOVERY_FAILED P1 권고 사항이 이 단계에 해당합니다:

세부 단계조치tkadmin 도구
억제페일오버/우회 방안 활성화서비스 관리 → 서비스 중지
근절근본 원인 특정·제거시스템 로그, 서비스 로그 분석
복구서비스 재시작·정상화 검증서비스 관리 → 서비스 시작
증거 수집로그·상태 정보 보존시스템 로그 → 내보내기

단계 4: 사후 활동 (Post-Incident Activity)​

  • PIR (Post-Incident Review): 인시던트 원인·대응·개선점 검토 회의
  • 레슨런드(Lessons Learned): 재발 방지 절차 문서화
  • MTTR 분석: 대응 시간 측정·보고
  • watchdog.yml 업데이트: 재발 방지를 위한 임계값 조정

3.3 보안 인시던트 지표 (IOC)​

RECOVERY_FAILED와 함께 아래 지표가 동반될 경우 보안 침해를 의심합니다:

지표의미
바이너리 해시 변경악성코드 주입 가능성
설정 파일 무단 변경내부자 위협 또는 외부 침입
비정상적 포트 오픈백도어 설치 가능성
디스크 풀로그 삭제 시도 또는 랜섬웨어
위험

IOC가 감지되면 즉시 ISO/IEC 27035에 따라 보안팀에 통보하고 포렌식 절차를 개시하십시오.

4. ISO/IEC 27035 — 정보보안 인시던트 관리​

ISO/IEC 27035는 정보보안 인시던트 관리에 대한 국제 표준입니다. RECOVERY_FAILED 에스컬레이션 경고에서 참조됩니다.

4.1 인시던트 심각도 분류 (Class 1~4)​

등급설명tkadmin 이벤트
Class 1경미 — 서비스 영향 없음SERVICE_RESTART
Class 2보통 — 부분 서비스 저하CONFIG_UPDATE
Class 3심각 — 서비스 중단FAILURE, RECOVERY_FAILED
Class 4위기 — 광범위한 시스템 침해RECOVERY_FAILED + IOC 동반
정보

RECOVERY_FAILED는 기본적으로 Class 3으로 분류됩니다. 바이너리/설정 변조 의심 시 Class 4로 상향하여 보안팀 개입이 필요합니다.

4.2 CIA 트라이어드 영향 평가​

정보보안의 3대 원칙(기밀성·무결성·가용성)에 대한 영향을 평가합니다:

원칙영문의미관련 이벤트
기밀성Confidentiality인가된 사용자만 데이터 접근CONFIG_UPDATE (설정 노출)
무결성Integrity데이터·시스템 변조 방지RECOVERY_FAILED (바이너리 변조 의심)
가용성Availability서비스가 필요 시 사용 가능FAILURE, RECOVERY_FAILED

5. ISO/IEC 20000-1:2018 — IT 서비스 관리​

ISO/IEC 20000-1:2018은 IT 서비스 관리 시스템(SMS)에 대한 국제 표준입니다. FAILURE 및 SERVICE_RESTART 권고 사항에서 참조됩니다.

5.1 주요 관련 조항​

조항내용tkadmin 연계
8.6 인시던트 관리인시던트 기록·분류·해결·종료알림 보고 + 읽음 관리
8.7 서비스 요청 관리표준화된 서비스 요청 처리tkcli 보고 통합
8.8 문제 관리반복 인시던트 근본 원인 제거RECOVERY_FAILED 후 PIR
8.5 가용성 관리SLA 기반 서비스 가용성 목표MTTR, MTBF 측정

5.2 서비스 복구 시간 목표 (RTO)​

서비스 등급RTO (복구 시간 목표)RPO (복구 시점 목표)
Tier 1 (핵심)4시간 이내1시간
Tier 2 (중요)8시간 이내4시간
Tier 3 (일반)24시간 이내24시간
팁

TACHYON 서비스(MariaDB, OpenSearch, Redis)는 일반적으로 Tier 1으로 분류됩니다. FAILURE P1 알림 발생 시 4시간 내 복구를 목표로 대응하십시오.

6. ITIL v4 Change Enablement​

변경 관리(Change Enablement)는 ITIL v4의 핵심 실천 사항 중 하나입니다. CONFIG_UPDATE 및 SERVICE_RESTART 권고 사항에서 참조됩니다.

6.1 변경 유형​

유형설명승인 절차tkadmin 이벤트
표준 변경사전 승인된 저위험 변경사전 등록으로 즉시 실행SERVICE_RESTART
일반 변경CAB 검토 및 승인 필요변경 요청서 제출 → CAB 승인CONFIG_UPDATE
긴급 변경즉각 실행 후 사후 승인ECAB(긴급 CAB) 사후 검토장애 복구 중 변경

6.2 변경 관리 7단계 (ITIL v4)​

1. 변경 요청(RFC) 등록
2. 변경 영향·위험 평가
3. CAB 승인
4. 변경 실행 (스케줄 조정)
5. 변경 검증·테스트
6. 변경 종료 및 기록
7. 변경 사후 검토 (PIR)
경고

CONFIG_UPDATE 알림이 발생했으나 변경 요청서(RFC)가 없는 경우, 무단 변경 가능성을 즉시 조사하십시오.

7. ITIL v4 Event Management​

이벤트 관리(Event Management)는 서비스 인프라에서 발생하는 모든 변화를 감지·평가·대응하는 실천 사항입니다.

7.1 이벤트 유형 분류​

유형설명조치 수준tkadmin 이벤트
정보성정상 운영 범위 내 이벤트로깅만SERVICE_RESTART
경고임계값 도달, 주의 필요모니터링 강화RECOVERY_SUCCESS
예외서비스 비정상, 즉각 대응 필요인시던트 처리FAILURE, RECOVERY_FAILED

7.2 이벤트 → 인시던트 전환 기준​

tkadmin에서 이벤트가 P1 인시던트로 전환되는 기준:

이벤트 유형 = FAILURE                    → P1 인시던트 즉시 생성
이벤트 유형 = RECOVERY_FAILED           → P1 인시던트 즉시 생성
24시간 내 동일 서비스 FAILURE 2회 이상  → P1 에스컬레이션
MTTR > 4시간                             → P1 SLA 위반 → 에스컬레이션

8. 에스컬레이션 경로 설계​

8.1 P1/P2 에스컬레이션 매트릭스​

경과 시간P1 CriticalP2 High
T+0담당자 호출담당자 알림
T+15분팀 리더 에스컬레이션—
T+30분부서장 에스컬레이션팀 리더
T+1시간CTO/경영진 보고부서장
T+4시간SLA 위반 공식 보고경영진 보고

8.2 보안 에스컬레이션 (ISO/IEC 27035)​

RECOVERY_FAILED + IOC 동반 시:

1. 즉시 해당 시스템 네트워크 격리 (억제)
2. 보안팀 즉시 통보 (CISO/보안 담당자)
3. 포렌식 증거 보존 (로그, 메모리 덤프)
4. 침해 범위 파악 (영향받은 시스템 목록화)
5. 법적 의무 검토 (개인정보 유출 시 72시간 내 신고)

9. 주요 용어 정리​
