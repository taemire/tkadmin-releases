3.2 Service Management (service)​

CommandFunctionKey Flags (Options)
checkCheck all service statusesNone
startStart a service[service] : Service name (all if omitted)
stopStop a service[service] : Service name (all if omitted)
restartRestart a service[service] : Service name (all if omitted)
jvmJVM memory optimization--set : Setting value (auto, S, M, L, 4G, etc.)
--dry-run : Simulation
--no-restart : Skip restart
loglevelView/set log level[service] : Target service
--set : Set level (INFO, DEBUG, etc.)
logrotateLog rotation management--set : Auto-apply recommended settings

Checks and controls the status of TACHYON and middleware services. Each subcommand (check, start, stop, restart, loglevel, jvm, logrotate) supports independent help (-h).

Usage​

tkctl service [action] [service_name] [flags]

Available Actions​

  • check: Check status of all services
  • start <service>: Start a service
  • stop <service>: Stop a service
  • restart <service>: Restart a service
  • jvm: JVM heap memory configuration management and auto-optimization
  • loglevel: View/set log level
  • logrotate: Log rotation management

Service List​

Middleware Services:

  • mariadb - MariaDB database
  • redis - Redis cache
  • nginx - NGINX web server
  • zookeeper - Zookeeper
  • kafka - Kafka message broker
  • opensearch - OpenSearch search engine
  • opensearch-dashboards - OpenSearch Dashboards
  • logstash-kafka-os - Logstash

TACHYON Services:

  • TACHYON-Api1 - API server
  • TACHYON-Auth1 - Authentication server
  • TACHYON-Manager1 - Management server
  • TACHYON-Stat1 - Statistics server
  • TACHYON-Report1 - Report server
  • TACHYON-Batch1 - Batch server
  • TACHYON-Watchdog1 - Watchdog server

Special Options:

  • all - All services (sequential start/stop)

Examples​

# Check all service statuses
tkctl service check

# Start a specific service
tkctl service start mariadb
tkctl service start TACHYON-Api1

# Start all services (sequential execution workflow)
tkctl service start all

/* Execution Order Details */

1. Level 1: Base Infrastructure (Infra)
   - zookeeper
   - redis
   - mariadb
   - opensearch
   - opensearch-dashboards
   (3-second wait)

2. Level 2: Messaging (Kafka)
   - kafka
   (5-second wait & connection status verification)

3. Level 3: Log Collection and Web Server
   - logstash-kafka-os
   - nginx

4. Level 4: TACHYON API
   - TACHYON-Api* (all discovered instances: Api1, Api2...)
   (60-second wait - API initialization)

5. Level 5: TACHYON Service Logic
   - TACHYON-Auth*
   - TACHYON-Manager*
   - TACHYON-Stat*
   - TACHYON-Report*
   - TACHYON-Batch*
   (10-second wait)

6. Level 6: Watchdog
   - TACHYON-Watchdog*


# Stop a specific service
tkctl service stop nginx

# Stop all services (reverse workflow)
tkctl service stop all
(Safely shuts down in order: Watchdog -> Service Logic -> API -> Logs/Web -> Kafka -> Infra)

3.2.1 Service Log Level Check and Configuration (loglevel)​

Checks or changes the logging level (DEBUG, INFO, WARN, ERROR, etc.) of each running service. Analyzes configuration files of Java-based TACHYON services and major middleware (Nginx, Redis, etc.) to display the currently applied level, and allows immediate changes through the --set option.

Usage:

# Check logging level of all services
tkctl service loglevel

# Check logging level of a specific service
tkctl service loglevel api

# Change logging level of a specific service
tkctl service loglevel api --set DEBUG
tkctl service loglevel auth --set INFO
tkctl service loglevel batch --set WARN

Supported Levels:

  • DEBUG, INFO, WARN, ERROR, TRACE, OFF

Diagnosis Target Files and Details:

  • TACHYON Services: *.yml_dev (plaintext originals), logback-spring.xml, log4j2.xml
    • Displays system logging level based on logging.level.* settings.
  • NGINX: nginx/conf/nginx.conf
    • Error Log: Extracts the level (DEBUG, INFO, ERROR, etc.) set in the error_log directive.
    • Access Log: Since NGINX access logs have no level control and only manage recording status, parses the access_log off; setting to display as ON/OFF.
  • Redis: redis.conf (loglevel directive)
    • Extracts Redis standard levels (DEBUG, VERBOSE, NOTICE, WARNING).

Log Level Change Logic (Java Services - v0.4.22 Enhancement):

StepActionDescription
1Modify *.yml_dev fileBatch change all keys under logging.level.* (root, com.tachyon, etc.)
2Copy *.yml fileReflect in the actual configuration file used by Spring Boot
3Sync *_ORIGIN fileAlso update the default configuration file (settings preserved after reinstall)
4Restart servicesystemctl restart TACHYON-[Svc]1.service

Note: The logging.level.root key is automatically created for services that don't have it (batch, watchdog, etc.).

3.2.2 Service Log Rotation Status Check and Configuration (logrotate)​

Checks the log rotation configuration status of TACHYON and middleware services and automatically configures missing settings.

Usage:

# Check all service log rotation status
tkctl service logrotate

# Check log rotation status of a specific service
tkctl service logrotate nginx

# Auto-configure log rotation for services with missing settings
tkctl service logrotate --set

# Configure log rotation for a specific service
tkctl service logrotate redis --set

Supported Services:

  • Middleware: nginx, redis, opensearch (main), opensearch-gc (GC logs), kafka, logstash, mariadb
  • TACHYON: api, auth, manager, stat, batch, report, watchdog (Java/Log4j2 based)
  • Management Tools: tkadmin, tkctl

Rotation Types:

  • logrotate: Uses the system logrotate daemon (/etc/logrotate.d/)
  • log4j2: Uses Java Log4j2 internal rolling policy
  • internal: Uses service's built-in rotation mechanism

3.2.3 Service JVM Memory Optimization and Management (jvm)​

Views, changes, or auto-optimizes JVM heap memory (-Xms, -Xmx) settings for TACHYON and related middleware based on system specifications.

Usage:

# Check all JVM service memory settings and system allocation
tkctl service jvm

# Change memory settings for a specific service
tkctl service jvm api --set 2G

# Apply official recommended presets (XS, S, M, L)
tkctl service jvm --set S

# Auto-optimize based on system specs and agent count (recommended)
tkctl service jvm --set auto

# Auto-optimize with explicit agent count
tkctl service jvm --set auto --agents 5000

# Preview changes without applying (Dry-run)
tkctl service jvm --set auto --dry-run

# Skip automatic service restart after configuration change
tkctl service jvm --set auto --no-restart

Auto-Optimization Algorithm (auto):

  • Analyzes the server's physical memory (RAM) size and the number of managed agents (Scale) to calculate optimal ratios per service.
  • Reflects the importance and actual load distribution of each component: OpenSearch (20%), Kafka (10%), API (5%), etc.
  • Ensures system stability with per-service minimum heap (256MB~1GB) guarantees and maximum cap (16GB) policies.

Preset Types:

  • XS (Extra Small): For low-spec test environments with 8GB or less (ultra-lightweight settings)
  • S (Small): For 16GB recommended specifications
  • M (Medium): For 32GB recommended specifications
  • L (Large): For high-spec environments with 64GB or more

Safety Features:

  • Automatically creates a backup (.bak) of the original file before any configuration change.
  • Resource Warning: When total JVM allocation exceeds 80% of physical memory, outputs a warning message and recommends manual review for operational stability.
  • Pre-reports to tkadmin during service restart to prevent Watchdog false alarms.
