3.4 Backup & Restore (backup/restore)​

Backs up and restores MariaDB and OpenSearch data. Implemented with Cobra native subcommand structure, providing seamless auto-completion and help support.

Usage​

# Backup
tkctl backup [mariadb|opensearch] [flags]

# Restore
tkctl restore [mariadb|opensearch] [args]

3.4.1 MariaDB Backup (backup mariadb)​

Uses mariabackup to back up the MariaDB database.

Usage:

# Full backup (default)
tkctl backup mariadb

# Specify backup type
tkctl backup mariadb --type full    # Full physical backup
tkctl backup mariadb --type incr    # Incremental backup
tkctl backup mariadb --type config  # Configuration data only (excludes logs)
tkctl backup mariadb --type log     # Full backup including logs

# Combined options
tkctl backup mariadb --type full --retention 30 --compress

Flags:

FlagShortDescriptionDefault
--type-tBackup type (config, log, full, incr)full
--retention-rRetention days (type-aware default — Log 90, Config 30, Full/Incr 30. Override when explicitly specified)type-aware
--compress-cUse qpress compression (type-aware default — Log forced true, Config false, Full/Incr true. Override when explicitly specified)type-aware
--incremental-basedirBase directory for incremental backup(auto-detected)

Output paths (per-type, BL-001 v0.6.42+):

--typeOutput directoryCompression (.qp)PrepareDefault retention
full${BACKUP_DIR}/full_$DATE/optional✅ (ready for restore)30 days
incr${BACKUP_DIR}/incr_$DATE/optional❌30 days
log${BACKUP_DIR}/weekly_log/$DATE/forced (qpress required)❌90 days
config${BACKUP_DIR}/daily_config/$DATE/default false❌30 days

Output example (--type log):

[ 16%] [0s] Preflight: checking backup environment...
[ 33%] [0s] Preparing: building mariabackup command...
[ 50%] [0s] Running: mariabackup --backup in progress...
[ 66%] [43s] Prepare: verifying backup consistency...
[ 83%] [43s] Metadata: saving backup info...
[100%] [43s] Cleanup: pruning old backups...
✅ MariaDB Backup (log) completed successfully.

  Size: 3.56 GB
  Duration: 43s

💡 Progress bar: Since v0.4.23, a visual progress bar is displayed in TTY environments. When output is piped (|) or redirected (>), the traditional text format is shown.

⚠️ --type log qpress requirement (v0.6.42+)

The Log type forces compression. If qpress is not found in the system PATH or in ${BaseDir}/mariadb/scripts/backup/bin/, the command fails immediately (typed error [1009] Log backup requires qpress compression but qpress binary was not found, exit code 1).

Operator actions:

  • Install via system package: dnf install qpress or apt install qpress
  • Or deploy to local path: ${BaseDir}/mariadb/scripts/backup/bin/qpress (must be chmod +x)
Configuration File Priority Principle

Authentication uses ${BaseDir}/conf/app_info.properties_dev's db_user / db_password as the single source of truth (SSOT). Environment variables MYSQL_USER / MYSQL_PASS / MYSQL_PORT are ignored (LogWarn) when they differ from the configuration file. tkcli's backup.env is not supported.

Multi-process file lock (v0.6.42+, BL-006)

When cron-driven backups and manual CLI invocations target the same BackupDir simultaneously, the second caller fails immediately with [1004] another backup is in progress (PID=N). Lockfiles are split per type — ${BackupDir}/.tkctl-mariadb-{full|incr|log|config}.lock — so different types may still run concurrently.

AspectLinux/macOSWindows
Mechanismsyscall.Flock(LOCK_EX|LOCK_NB) kernel advisory lockO_EXCL lockfile fallback (advisory only)
Auto-release✅ Released on fd close at process exit — no stale residue⚠️ Lockfile may persist after abnormal termination
Stale recoveryNot requiredIf the recorded PID is dead, the operator must remove it manually: rm ${BackupDir}/.tkctl-mariadb-*.lock

Diagnostic branches: a contention (other holder) returns [1004 BACKUP_IN_PROGRESS] + holder PID. A write-permission shortage on BackupDir (e.g. an unprivileged user invoking against a root-owned directory) returns [1005 INVALID_BACKUP_DIR] + operator guidance (BackupDir write permission denied — run as root or with sufficient privileges), so first-line triage is unambiguous.

Operational guidance: distribute cron schedules to avoid temporal overlap with manual CLI runs. Even when they collide, the second call is blocked instead of advancing into database lock contention or backup corruption. In production, run tkctl backup mariadb with root privileges.

Minimal side effects

--dry-run only creates a transient .preflight-probe-<PID> file (immediately removed) in BackupDir and acquires/releases the lockfile once. If another backup of the same type is in progress, the run reports [FAIL] lockfile-available immediately and recommends a retry.

--with-connect — DB credential accuracy ping (BL-007 follow-up)​

The default --dry-run only verifies that credentials are loaded (the credentials check). A wrong password still reports as "set (N chars)" / OK. Pass --with-connect to add a SELECT 1 ping against the live mariadb server and verify accuracy as well.

$ tkctl backup mariadb --dry-run --with-connect --type log

Non-localhost / Unix socket examples:

# Remote mariadb (e.g. read replica)
$ tkctl backup mariadb --dry-run --with-connect --connect-host 192.168.1.100 --type log

# Unix socket (performance-tuned environments)
$ tkctl backup mariadb --dry-run --with-connect --connect-socket /var/run/mysqld/mysqld.sock --type log
host / socket are mutually exclusive

Specifying both --connect-host and --connect-socket is rejected immediately (mutually exclusive). When socket is set, host is ignored and the diagnostic target is reported as unix:<path>. If the socket file is missing, the run reports [FAIL] db-connect-ping ... Unix socket 미존재 before invoking the driver, so first-line triage is unambiguous.

ScenarioResult
No credentials[SKIP] db-connect-ping (the credentials check already explains why)
Wrong password[FAIL] db-connect-ping + Access denied (with password redacted)
DB down / port closed[FAIL] db-connect-ping + connection refused or timeout
Valid credentials + live DB[OK] db-connect-ping
Host assumption

The DSN is hard-coded to tcp(127.0.0.1:<port>), assuming tkctl runs locally on the database server. Non-localhost mariadb or Unix-socket setups are out of scope (separate follow-up). --with-connect only takes effect under --dry-run; in a real backup, mariabackup itself enforces credential validity.

3.4.2 OpenSearch Backup (backup opensearch)​

Creates an index snapshot in the OpenSearch snapshot repository.

Usage:

# Initialize snapshot repository (one-time setup)
tkctl backup opensearch --init-repo

# Create snapshot (auto-named)
tkctl backup opensearch

# Create snapshot with specified name
tkctl backup opensearch snap_20260116

Flags:

FlagDescription
--init-repoInitialize snapshot repository (fs type). ⚠️ If path.repo is missing in opensearch.yml, the OpenSearch service is automatically restarted (~30s downtime)
--usernameOpenSearch username (default: auto-detect)
--passwordOpenSearch password (default: auto-detect)
⚠️ --init-repo Auto-Restart Policy (v0.6.39)

When --init-repo runs and path.repo is missing from opensearch.yml, the following steps are performed in a single command:

  1. Auto-append path.repo to opensearch.yml
  2. Display warning banner + 3-second grace period (Ctrl+C to abort)
  3. Auto-execute systemctl restart opensearch
  4. Poll /_cluster/health and wait up to 90 seconds for cluster readiness
  5. Register the snapshot repository

OpenSearch connectivity is briefly interrupted (~30s) during the restart. Run during a maintenance window or coordinate with the operations team. If path.repo is already configured, no restart occurs.

Output example (path.repo missing → auto-restart case):

2026-04-13 19:10:02 [INFO] Initializing OpenSearch Snapshot Repository 'backup'...
2026-04-13 19:10:02 [WARN] path.repo not configured in opensearch.yml — auto-appending.
2026-04-13 19:10:02 [SUCC] path.repo appended to opensearch.yml.
2026-04-13 19:10:02 [WARN] ========================================================================
2026-04-13 19:10:02 [WARN] ⚠️  Auto-restarting OpenSearch service (required for path.repo)
2026-04-13 19:10:02 [WARN]     → OpenSearch connectivity will be interrupted briefly (~30s)
2026-04-13 19:10:02 [WARN]     → Press Ctrl+C within 3s to abort
2026-04-13 19:10:02 [WARN] ========================================================================
2026-04-13 19:10:05 [INFO] Executing: systemctl restart opensearch
2026-04-13 19:10:08 [SUCC] OpenSearch restart issued — waiting for cluster readiness...
2026-04-13 19:10:25 [SUCC] OpenSearch is ready — proceeding with repository registration.
2026-04-13 19:10:25 [SUCC] Repository initialized successfully.

Output example (normal snapshot creation):

2026-01-16 10:35:00 [INFO] Creating OpenSearch snapshot...
2026-01-16 10:35:00 [INFO] Snapshot name: snap_20260116_103500
2026-01-16 10:35:10 [SUCC] Snapshot created successfully.

3.4.3 MariaDB Restore (restore mariadb)​

Restores the MariaDB database from a mariabackup backup.

Usage:

# Restore from full backup
tkctl restore mariadb /backup/full_20260116

# Restore from full + incremental backups
tkctl restore mariadb /backup/full_20260116 /backup/incr_20260117 /backup/incr_20260118

Output example:

2026-01-16 11:00:00 [INFO] Starting MariaDB restore...
2026-01-16 11:00:00 [INFO] Full backup: /backup/full_20260116
2026-01-16 11:00:00 [WARN] MariaDB service is running. Please stop it before restoring.
2026-01-16 11:00:00 [INFO] Command: systemctl stop mariadb

Caution: The MariaDB service must be stopped before restoring.

3.4.4 OpenSearch Restore (restore opensearch)​

Restores OpenSearch indices from a saved snapshot.

Usage:

tkctl restore opensearch snap_20260116

Output example:

2026-01-16 11:10:00 [INFO] Starting OpenSearch snapshot restore...
2026-01-16 11:10:00 [INFO] Snapshot: snap_20260116
2026-01-16 11:10:15 [SUCC] Snapshot restore completed.
