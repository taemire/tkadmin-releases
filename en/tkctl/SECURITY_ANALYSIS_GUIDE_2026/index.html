Technical Vulnerability Analysis & Assessment Guide for Critical Information and Communication Infrastructure (2026)

This document is an analysis guide written to implement Linux/Unix server security scanning features for the tkctl project, based on the [2026 Detailed Guide for Technical Vulnerability Analysis and Assessment of Critical Information and Communication Infrastructure] published by KISA (Korea Internet & Security Agency).

1. Overview and Purpose​

This guide defines detailed criteria for analyzing and assessing technical vulnerabilities that threaten the stable operation of critical information and communication infrastructure. tkctl aims to automatically diagnose security configurations of Linux servers based on these criteria and provide remediation guides for vulnerable items.

2. Scan Targets and Classification System (Unix/Linux Servers)​

Linux server security scan items are classified into 5 major domains, with each item identified by a code in the format U-XX (abbreviation for Unix).

CategoryItem Code RangeKey Check Areas
1. Account ManagementU-01 ~ U-13Root remote access restriction, password complexity/length/expiration, account lockout, UID management
2. File and Directory ManagementU-14 ~ U-35Critical config file owner/permissions, SUID/SGID, world writable, UMASK
3. Service ManagementU-36 ~ U-63Disable unnecessary services, crontab, NFS, RPC, DNS, SSH management
4. Patch ManagementU-64Verify latest security patches are applied
5. Log ManagementU-65 ~ U-67Regular log review, syslog configuration, log retention

Note: The item numbering system was expanded in the KISA 2026 guide (67 items total). This document focuses on items most relevant to the TACHYON environment.

Extended Scan Targets: Web Services + DBMS​

The KISA 2026 guide also includes Web Services (WEB-xx) and DBMS (D-xx) domains in addition to Unix servers. Since TACHYON uses Nginx + MariaDB, these items are also subject to inspection.

CategoryItem Code RangeKey Check AreasTACHYON Relevance
Web ServicesWEB-01 ~ WEB-26Account/Service/Security settings/Patch/LogNginx + tkadmin Web UI
DBMSD-01 ~ D-26Account/Access/Options/Patch managementMariaDB

3. Unix/Linux Server Check Items Detail (U-01 ~ U-67)​

Written based on the KISA 2026 guide standards. TACHYON environment-specific considerations (Rocky Linux 9 + Nginx + MariaDB + OpenSearch + Kafka + Redis) are reflected in each item.

1. Account Management​

U-01. Restrict Remote Access to Root Account​

FieldDetails
SeverityHigh
Check TargetTACHYON server (Rocky Linux 9) SSH configuration. tkadmin requires SSH access
CompliantRemote terminal service is not used, or root direct access is blocked when in use
VulnerableRoot direct access is allowed when remote terminal service is in use

Check Method:

# Step 1) Check SSH configuration file
cat /etc/ssh/sshd_config | grep PermitRootLogin
# Step 2) Set PermitRootLogin value to no
vi /etc/ssh/sshd_config
# Step 3) Restart SSH service
systemctl restart sshd

tkctl Implementation Note: Parse the PermitRootLogin value in /etc/ssh/sshd_config to verify it is set to no

U-02. Password Complexity Settings​

FieldDetails
SeverityHigh
Check TargetTACHYON server and MariaDB/Redis account password complexity
CompliantMinimum password length of 8 characters with mandatory letters, numbers, and special characters is configured
VulnerableMinimum password length of 8 characters with mandatory letters, numbers, and special characters is not configured

Check Method:

# Step 1) Check password complexity configuration file
cat /etc/security/pwquality.conf
cat /etc/pam.d/system-auth | grep pam_pwquality.so
cat /etc/pam.d/password-auth | grep pam_pwquality.so
# Step 2) Modify /etc/security/pwquality.conf file
vi /etc/security/pwquality.conf

tkctl Implementation Note: Parse /etc/security/pwquality.conf -- check minlen, dcredit, ucredit, lcredit, ocredit options

U-03. Account Lockout Threshold Settings​

FieldDetails
SeverityHigh
Check TargetTACHYON server SSH/PAM account lockout settings
CompliantAccount lockout threshold is set to 5 attempts or fewer
VulnerableAccount lockout threshold is not set, or is set to more than 5 attempts

Check Method:

# Step 1) Check account lockout threshold configuration file
cat /etc/pam.d/system-auth | grep pam_faillock.so
cat /etc/pam.d/password-auth | grep pam_faillock.so
# Step 2) Modify /etc/pam.d/system-auth and /etc/pam.d/password-auth files
vi /etc/pam.d/system-auth
vi /etc/security/faillock.conf

tkctl Implementation Note: Check the deny value of pam_faillock.so in /etc/pam.d/system-auth. RHEL 8+: /etc/security/faillock.conf

U-04. Password File Protection​

FieldDetails
SeverityHigh
Check TargetShadow password usage (enabled by default on Rocky Linux 9)
CompliantShadow passwords are used, or passwords are stored encrypted
VulnerableShadow passwords are not used, and passwords are not stored encrypted

# Step 2) Convert to shadow password if not in use

tkctl Implementation Note: Check if the 2nd field of /etc/passwd is x (shadow usage)

U-05. Prohibit UID '0' for Accounts Other Than Root​

FieldDetails
SeverityMedium
Check TargetUID verification for TACHYON service accounts (tachyon, opensearch, etc.)
CompliantNo accounts other than root have the same UID as root
VulnerableAccounts other than root have the same UID as root

Check Method:

# Step 1) Check for accounts with UID 0 in /etc/passwd

# Step 2) Change UID of non-root accounts with UID 0

tkctl Implementation Note: Parse /etc/passwd -- check if any accounts other than root have UID=0

4. tkctl Implementation Considerations​

  1. OS Compatibility:

    • Configuration file path differences between RHEL (CentOS, Rocky) and Debian (Ubuntu) families must be considered. (e.g., pam configuration paths)
    • The tkctl internal logic must first identify the OS type (ProcessOSProbe) and then branch accordingly.

  2. Scan Logic:

    • Simple Permission Check: File mode and owner (Uid) inspection using os.Stat.
    • Content Parsing: Configuration file key-value pair parsing using bufio.Scanner, etc. Lines with comments (#) must be ignored.

  3. Result Reporting:

    • Scan results must include Code, Name, Status (Compliant/Vulnerable/N/A), and Detail (current value) fields.
    • Output in JSON format for integration with visualization tools.
