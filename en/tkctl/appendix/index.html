Appendix A: Service Logging Level Diagnostic Technical Specification

This appendix describes in detail the configuration file locations and parsing logic that the tkctl service loglevel command references to diagnose each service's logging level. System engineers can refer to this guide to understand the basis of information reported by tkctl and manually verify settings when necessary.

1. Overview​

tkctl directly parses each service's configuration files as text or uses a YAML parser to extract logging-related directives. Each service searches for relative paths based on the installation base directory (${BaseDir}).

2. Middleware Services​

ServiceReference File PathDiagnostic Method (Parsing Logic)
NGINXnginx/conf/nginx.conf1. Error Log: Extracts the last argument of the error_log directive to display the level (DEBUG, INFO, etc.).
2. Access Log: Since NGINX does not use a level concept for access logs, it only checks for the presence of the access_log off; directive and reports as ON/OFF status.
Display format: Level (Access: ON/OFF)
Redisredis/conf/redis.conf or redis/redis.confExtracts the right-side value of the loglevel directive.
Display example: loglevel notice → NOTICE
MariaDBmariadb/my.cnfOnly checks file existence and displays the default value INFO (Default).
(MariaDB primarily uses detailed logging settings rather than a single level setting)

3. TACHYON Java Services​

All TACHYON Java services (api, auth, manager, stat, batch, report, watchdog) search for files according to the following priority.

3.1 Configuration File Search Priority​

  1. [svc]/conf/[svc].yml_dev (YAML) - Highest priority (Plaintext original)
  2. [svc]/conf/[svc].yml (YAML)
  3. [svc]/conf/logback-spring.xml (XML)
  4. [svc]/conf/log4j2.xml (XML)
  5. [svc]/conf/application.yml (YAML)

3.2 Parsing Technical Specification​

YAML Parsing (.yml, .yml_dev)​

Uses the gopkg.in/yaml.v3 library to extract values from the following hierarchy:

  • Path: logging > level > root
  • Structure example: 
    logging:
      level:
        root: INFO  # <-- extraction target

XML Parsing (.xml)​

Extracts the level attribute of the <root> tag through string pattern matching:

  • Regex/Pattern: <root level="([^"]+)">
  • Structure example: 
    <root level="DEBUG">  <!-- <-- level attribute value extracted -->
        <appender-ref ref="CONSOLE" />
    </root>

4. Management Tools​

ServiceReference File PathDiagnostic Method (Parsing Logic)
tkadmin${BaseDir}/tkadmin.ymlExtracts the logging.level.root value through YAML parsing.
tkctl${BaseDir}/tkctl.iniChecks the log_level key value in the configuration file. (INFO if not set)

5. Diagnostic Status Descriptions​

StatusMeaningRecommended Action
OKConfiguration file found and parsed successfully-
CONFIG NOT FOUNDConfiguration file does not exist at the specified pathVerify the installation path of the service.
INVALIDFile was found but logging level directive is missingCheck whether the root level setting is missing in the configuration file.
-Diagnosis not possibleSpecify a particular service for detailed diagnosis.

Note: For Java services, tkctl recommends a management workflow of modifying the yml_dev file and copying it to yml for application.

6. System Environment Configuration Diagnostic and Setup Specification (env)​

The tkctl env command diagnoses and optimizes kernel and system environment variables essential for TACHYON operations.

6.1 SELinux​

CategoryDiagnostic/Reference Logic
Current execution modePrioritizes executing the getenforce command; if unavailable, extracts Current mode from sestatus output.
Permanent configuration modeDirectly parses the SELINUX= directive value from /etc/selinux/config file.
Change processWhen --set is executed, performs both immediate application via setenforce and permanent application via config modification simultaneously. (Provides reboot guidance when setting to Disabled)

6.2 System Resource Limits (Ulimit)​

CategoryDiagnostic/Reference Logic
Kernel-wide limitsParses the /etc/security/limits.conf file to collect nofile settings by domain.
Per-service limitsQueries the actual values assigned at startup through systemctl show -p LimitNOFILE --value [SVC] for systemd units.
TACHYON blockWhen configuring via the --set feature, manages a dedicated block using # --- TACHYON LIMITS START --- markers to prevent corruption of existing files.

6.3 Firewall​

  • Active status detection: Simultaneously checks whether firewalld and iptables services are running via systemctl is-active.
  • Port list extraction: When firewalld is active, parses and provides results from firewall-cmd --list-ports and --list-services.
  • Auto-allow logic: When the --set argument is omitted, automatically detects and opens the TACHYON management port (443) and the currently active SSH port.

6.4 SSH Configuration and Integration​

  • Port detection: Searches for uncommented Port directives in the /etc/ssh/sshd_config file. (Assumes standard port 22 if directive is absent)
  • Change and integration workflow:
    1. Modify the Port number in /etc/ssh/sshd_config
    2. Permanently allow the new port through the firewall via firewall-cmd --add-port=[NEW_PORT]/tcp (--permanent)
    3. Execute firewall-cmd --reload to immediately apply firewall settings
    4. Execute systemctl restart sshd to complete the configuration

7. JVM Memory Configuration and Auto-Optimization Technical Specification (service jvm)​

The tkctl service jvm command follows this technical specification to diagnose and optimize heap memory for JVM-based services.

7.1 Per-Service Configuration Files and Parsing Logic​

ServiceConfiguration File PathExtraction/Modification Logic
OpenSearchopensearch/opensearch/config/jvm.optionsDirectly parses lines starting with -Xms, -Xmx.
Kafkakafka/bin/kafka-server-start.shExtracts the KAFKA_HEAP_OPTS environment variable assignment line using regex.
Logstashlogstash/logstash-kafka-os/config/jvm.optionsDirectly parses -Xms, -Xmx lines.
TACHYON Services[svc]/bin/tachyon-[svc].shExtracts -Xms, -Xmx patterns from java command arguments in the execution script.
Watchdogbin/tachyon-watchdog.shUses the same shell script regex parsing method.

7.2 Auto-Optimization (auto) Algorithm Details​

When the --set auto option is executed, system total memory is allocated based on the following weight model.

  1. Optimization formula: Allocation = (Total Memory - Base OS Occupancy) * Per-Service Weight
  2. Per-service weights:
    • opensearch: 20%
    • kafka: 10%
    • api: 5%
    • manager, auth, stat, batch, report: 3~5% each
    • watchdog: 2% (minimum 128MB guaranteed)
  3. Guardrails:
    • Maximum cap: Single service heap size is capped at 16GB.
    • Minimum floor: Minimum floors of 256MB~1GB (varies by service) are applied to ensure service visibility.
    • Agent count weighting: Higher --agents values dynamically increase the proportion of API and Manager services.

7.3 Safety Management Process​

  • Backup policy: Creates a ${ConfigFile}.bak file before changing settings. If a backup already exists, it uses timestamps to protect the history instead of overwriting.
  • Resource warning: If total JVM allocation exceeds 80% of physical memory, a warning message is output and manual review is recommended for operational stability.

8. OS Memory Management and Occupancy Technical Specification (Memory Management)​

This section provides the technical basis for the phenomenon where memory usage reported by the tkctl info command appears higher than what users perceive.

8.1 Linux vs Windows Memory Management Philosophy Differences​

ItemLinux (Operational Environment)Windows (Comparison Environment)
Core philosophy"Free RAM is wasted RAM"User-visible resource-oriented flexible allocation
Cache utilizationDedicates all remaining memory to Page Cache for disk I/O performance improvementPerforms prefetching via Superfetch, but tends to count as 'available' in reporting
Usage calculationUsed = Total - Free (Free is purely unused physical space)Used = Total - Available (includes immediately available cache as available)

8.2 Page Cache and Over-Occupancy​

  1. Over-occupancy phenomenon (Memory Occupancy):

    • The Linux kernel retains file data in physical memory (Page Cache) when reading or writing files. This allows subsequent access to the same file to respond immediately from fast memory instead of slow disk.
    • TACHYON services (OpenSearch, MariaDB, etc.) continuously process large amounts of data, so cache occupancy increases over time and Free memory may display as close to 0.

  2. User visibility:

    • tkctl info directly reports the physical occupancy state of the OS kernel, so it includes all cache areas as 'in use'.

  3. Actual availability (Available Memory):

    • This cache area is reclaimable space that the OS can immediately free when an application requests actual memory allocation. Therefore, even if the occupancy percentage is high, it does not affect the system's practical work capacity.

8.3 Conclusion and Assessment Criteria​

The high memory occupancy shown in tkctl info is mostly the normal result of Linux kernel optimization for I/O performance improvement. To determine whether system resources are actually insufficient, use the following indicators rather than occupancy percentages.

  • Swap occurrence: Check whether physical memory shortage is causing frequent disk compression/swapping.
  • OOM Killer logs: Check kernel logs (dmesg) for process forced termination history due to memory exhaustion.

TIP: To know the practically 'available' amount in a Linux environment, refer to the Available metric rather than the Free (idle) metric. (tkctl plans to include the available metric in future versions.)

9. Log Analysis System Technical Specification (analyze)​

This appendix describes in detail the internal operating principles and query structures used by the tkctl analyze command to analyze log data from MariaDB and OpenSearch. System engineers can refer to this guide to verify tkctl analysis results or perform manual analysis directly without the CLI tool.

9.1 Overview and Data Sources​

The tkctl analyze command queries two data sources from the TACHYON system:

Data SourcePurposeConnection Information Source
OpenSearchReal-time log aggregation, anomaly detectionapp_info.properties → opensearch_ip, opensearch_port, opensearch_id, opensearch_pw
MariaDBLog table growth trend analysisapp_info.properties → db_ip, db_port, db_user, db_password

9.2 OpenSearch Index Naming Convention​

The TACHYON system stores media control event logs in monthly indices in OpenSearch.

Index name format:

{SPCODE}_{YEAR}_{MONTH}

Examples:

  • Organization code (SPCODE): 00133fkfkg
  • January 2026: 00133fkfkg_2026_01
  • December 2025: 00133fkfkg_2025_12

How to check SPCODE:

# Check from app_info.properties file
grep "spcode" /usr/local/TACHYON/TTS40/conf/app_info.properties_dev

# Or check database name (same value)
tkctl info

Manual index listing:

# Query directly via OpenSearch API
curl -s "http://localhost:9200/_cat/indices?v" | grep -E "00133fkfkg_"

9.3 OpenSearch Field Mapping​

Main field structure of the media control log index:

Field NameTypeDescriptionExample Value
server_dtkeywordServer reception date (YYYY-MM-DD)2026-01-15
client_dtkeywordClient event occurrence date2026-01-15
puidkeywordAgent unique identifier (UUID)a1b2c3d4-e5f6-7890-abcd-ef1234567890
process_nmkeyword/textExecution process pathC:\Windows\Explorer.EXE
file_nmkeyword/textTarget file/path accessedE:\Documents\report.xlsx
event_typekeywordEvent type codeREAD, WRITE, COPY

Note: Fields with the .keyword suffix are used for exact match searches and aggregations. In aggregation queries, be sure to use the process_nm.keyword form.

9.4 media-top: Top N Analysis by Process+Path​

Purpose: Identify which processes accessed which paths most frequently

9.4.1 OpenSearch DSL Query​

When executing tkctl analyze media-top --month 2026-01 --limit 20, the following DSL is used internally:

{
  "size": 0,
  "query": {
    "range": {
      "server_dt": {
        "gte": "2026-01-01",
        "lt": "2026-02-01"
      }
    }
  },
  "aggs": {
    "by_process_path": {
      "composite": {
        "size": 20,
        "sources": [
          { "process": { "terms": { "field": "process_nm.keyword" } } },
          { "path": { "terms": { "field": "file_nm.keyword" } } }
        ]
      }
    }
  }
}

9.4.2 Manual Execution Method​

# Execute query directly on OpenSearch
curl -X POST "http://localhost:9200/00133fkfkg_2026_01/_search" \
  -H "Content-Type: application/json" \
  -u "admin:admin" \
  -d '{
    "size": 0,
    "query": {
      "range": {
        "server_dt": { "gte": "2026-01-01", "lt": "2026-02-01" }
      }
    },
    "aggs": {
      "by_process_path": {
        "composite": {
          "size": 10,
          "sources": [
            { "process": { "terms": { "field": "process_nm.keyword" } } },
            { "path": { "terms": { "field": "file_nm.keyword" } } }
          ]
        }
      }
    }
  }' | jq '.aggregations.by_process_path.buckets'

9.4.3 Response Structure and Interpretation​

{
  "buckets": [
    {
      "key": {
        "process": "C:\\Windows\\Explorer.EXE",
        "path": ""
      },
      "doc_count": 42349
    },
    {
      "key": {
        "process": "C:\\Windows\\system32\\svchost.exe",
        "path": ""
      },
      "doc_count": 5966
    }
  ]
}
  • key.process: Execution process path
  • key.path: Target file/path accessed (empty string indicates events without a specified path)
  • doc_count: Number of occurrences for the combination

9.5 agent-anomaly: Anomalous Agent Detection​

Purpose: Identify agents logging excessively compared to the average

9.5.1 OpenSearch DSL Query​

{
  "size": 0,
  "query": {
    "range": {
      "server_dt": {
        "gte": "2026-01-01",
        "lt": "2026-02-01"
      }
    }
  },
  "aggs": {
    "by_agent": {
      "terms": {
        "field": "puid.keyword",
        "size": 10000
      }
    },
    "agent_stats": {
      "extended_stats_bucket": {
        "buckets_path": "by_agent._count"
      }
    }
  }
}

9.5.2 Key Aggregation Description​

Aggregation TypeFieldDescription
termsby_agentLog count aggregation per agent (PUID)
extended_stats_bucketagent_statsBucket statistics calculation (mean, standard deviation, etc.)

9.5.3 Anomaly Detection Algorithm​

tkctl uses the following logic to determine anomalous agents:

Anomaly = (Agent Log Count / Overall Average) >= Threshold (default 5.0)

Assessment criteria:

  • Critical: 10x or more above average (Multiplier >= 10.0)
  • Warning: 5x to 10x above average (Multiplier >= 5.0)

9.5.4 Manual Analysis Procedure​

# 1. Query per-agent statistics
curl -X POST "http://localhost:9200/00133fkfkg_2026_01/_search" \
  -H "Content-Type: application/json" \
  -u "admin:admin" \
  -d '{
    "size": 0,
    "aggs": {
      "by_agent": {
        "terms": { "field": "puid.keyword", "size": 10000 }
      },
      "agent_stats": {
        "extended_stats_bucket": { "buckets_path": "by_agent._count" }
      }
    }
  }' > /tmp/agent_stats.json

# 2. Check average value
jq '.aggregations.agent_stats.avg' /tmp/agent_stats.json

# 3. Check standard deviation
jq '.aggregations.agent_stats.std_deviation' /tmp/agent_stats.json

# 4. Identify top anomalous agents (filter for 5x or more above average)
AVG=$(jq '.aggregations.agent_stats.avg' /tmp/agent_stats.json)
THRESHOLD=$(echo "$AVG * 5" | bc)
jq --argjson th $THRESHOLD \
  '.aggregations.by_agent.buckets | map(select(.doc_count >= $th)) | sort_by(-.doc_count) | .[0:10]' \
  /tmp/agent_stats.json

9.7 recommend: Data-Driven Analysis Recommendations​

Purpose: Analyze growth rates of MariaDB log tables and automatically recommend items requiring in-depth analysis

9.7.1 MariaDB Query Structure​

Current month and previous month count query:

-- January 2026 count
SELECT COUNT(*)
FROM TPU_ISTAT_MEDIA_EVENT_LOG
WHERE server_dt >= '2026-01-01' AND server_dt < '2026-02-01';

-- December 2025 count (previous month)
SELECT COUNT(*)
FROM TPU_ISTAT_MEDIA_EVENT_LOG
WHERE server_dt >= '2025-12-01' AND server_dt < '2026-01-01';

9.7.2 Analysis Target Tables​

Table NameAnalysis TypeRecommended Command
TPU_ISTAT_MEDIA_EVENT_LOGMedia control logstkctl analyze media-top
TPU_ISTAT_EVENT_LOGEvent logstkctl analyze process-top
TPU_ISTAT_SYS_LOGSystem logstkctl analyze process-top

9.7.3 Growth Rate Calculation Formula​

Growth Rate (%) = (Current Month Count - Previous Month Count) / Previous Month Count x 100

9.7.4 Priority Assessment Criteria​

PriorityConditionDisplay IconRecommended Action
1 (Surge)Growth rate >= 100%Red indicatorImmediate analysis required
2 (Rapid increase)50% <= Growth rate < 100%Warning indicatorRoot cause analysis recommended
3 (Normal)Growth rate < 50%OK indicatorWithin normal range

9.7.5 Manual Analysis Script​

#!/bin/bash
# Manual execution of recommend analysis

DBNAME="00133FKFKG"
MONTH="2026-01"
PREV_MONTH="2025-12"

# Media control log growth rate analysis
CURRENT=$(mysql -u tachyon -p'password' $DBNAME -N -e \
  "SELECT COUNT(*) FROM TPU_ISTAT_MEDIA_EVENT_LOG
   WHERE server_dt >= '${MONTH}-01' AND server_dt < DATE_ADD('${MONTH}-01', INTERVAL 1 MONTH);")

PREVIOUS=$(mysql -u tachyon -p'password' $DBNAME -N -e \
  "SELECT COUNT(*) FROM TPU_ISTAT_MEDIA_EVENT_LOG
   WHERE server_dt >= '${PREV_MONTH}-01' AND server_dt < '${MONTH}-01';")

if [ "$PREVIOUS" -gt 0 ]; then
  RATE=$(echo "scale=1; ($CURRENT - $PREVIOUS) * 100 / $PREVIOUS" | bc)
  echo "Media control logs: Current=${CURRENT}, Previous=${PREVIOUS}, Growth rate=${RATE}%"

  if (( $(echo "$RATE >= 100" | bc -l) )); then
    echo "CRITICAL - Immediate analysis required: tkctl analyze media-top --month $MONTH"
  elif (( $(echo "$RATE >= 50" | bc -l) )); then
    echo "WARNING - Root cause analysis recommended: tkctl analyze media-top --month $MONTH"
  else
    echo "OK - Within normal range"
  fi
fi

9.8 db-trend: DB Data Growth Trend Analysis​

Purpose: Track cumulative growth trends of LOG tables by period (1 week, 1 month, 1 quarter, 1 year)

9.8.1 MariaDB Query Structure​

Cumulative count query by period:

-- Cumulative count up to 1 week ago (Start Total)
SELECT COUNT(*) FROM TPU_ISTAT_MEDIA_EVENT_LOG
WHERE server_dt < DATE_SUB(CURDATE(), INTERVAL 7 DAY);

-- Cumulative count up to current (End Total)
SELECT COUNT(*) FROM TPU_ISTAT_MEDIA_EVENT_LOG
WHERE server_dt < DATE_ADD(CURDATE(), INTERVAL 1 DAY);

9.8.2 Growth Rate Calculation​

Period Increase Count = End Total - Start Total
Growth Rate (%) = (End Total - Start Total) / Start Total x 100

9.8.3 Trend Assessment Criteria​

TrendGrowth Rate ConditionDisplay
Surge (Critical)>= 50%Red indicator
Rapid increase (Warning)20% ~ 50%Orange indicator
Gradual (Normal)5% ~ 20%Upward trend
Stable0% ~ 5%Flat
Decrease< 0%Downward trend

9.8.4 Manual Analysis Query Example​

-- Growth trend for the past week
SET @start_total = (SELECT COUNT(*) FROM TPU_ISTAT_MEDIA_EVENT_LOG
                    WHERE server_dt < DATE_SUB(CURDATE(), INTERVAL 7 DAY));
SET @end_total = (SELECT COUNT(*) FROM TPU_ISTAT_MEDIA_EVENT_LOG);
SET @increase = @end_total - @start_total;
SET @rate = IF(@start_total > 0, (@increase / @start_total) * 100, 100);

SELECT 'Past 1 week' AS period,
       @start_total AS start_cnt,
       @end_total AS end_cnt,
       @increase AS increase,
       CONCAT(ROUND(@rate, 1), '%') AS growth_rate,
       CASE
         WHEN @rate >= 50 THEN 'CRITICAL - Surge'
         WHEN @rate >= 20 THEN 'WARNING - Rapid increase'
         WHEN @rate >= 5 THEN 'Gradual (Normal)'
         WHEN @rate > 0 THEN 'Stable'
         ELSE 'Decrease'
       END AS trend;

9.11 Troubleshooting​

9.11.1 OpenSearch Connection Failure​

OpenSearch call failed: dial tcp 127.0.0.1:9200: connect: connection refused

Causes and remediation:

  1. Check OpenSearch service: systemctl status opensearch
  2. Check port listening: ss -tlnp | grep 9200
  3. Check firewall: firewall-cmd --list-ports

9.11.2 Index Not Found Error​

No data available for the specified period.

Causes and remediation:

  1. Verify index exists: curl "http://localhost:9200/_cat/indices?v" | grep {spcode}
  2. Check date range: Verify that data exists for the specified month
  3. Check SPCODE: Confirm the organization code in the index name is correct

9.11.3 MariaDB Connection Failure​

DB connection failed: dial tcp 127.0.0.1:3306: connect: connection refused

Causes and remediation:

  1. Check MariaDB service: systemctl status mariadb
  2. Verify connection info: db_ip, db_port, db_user, db_password in app_info.properties
