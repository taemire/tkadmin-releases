12. 알림 보고

12.1 개요​

알림 보고 화면은 Watchdog 엔진의 자동 장애 감지 이벤트와 tkctl 외부 CLI 도구를 통한 수동 작업 결과를 단일 테이블에 통합 표시합니다. 운영자는 이 화면을 통해 장애 발생·자동 복구·수동 제어 등 모든 운영 이벤트를 시간순으로 파악하고, 각 이벤트에 대한 ITIL v4 기반 우선순위 조치 권고 사항을 확인할 수 있습니다.

12.2 화면 구성​

영역설명
탭실시간 장애 감지 현황 / 전체 조치 감사 로그
상단 툴바유형 필터, 검색어 입력, 미확인만 보기, 검색·읽음 처리·모두 읽음 버튼
보고 테이블시간, 처리 결과(유형), 서비스, 메시지, 확인 여부 컬럼

테이블 컬럼​

컬럼설명
시간이벤트 발생 시각 — 헤더 시각 표시 기준(내 시각/서버 시각/UTC)에 따라 통일 표시
처리 결과이벤트 유형 배지 (장애 발생·복구 성공·복구 실패·설정 변경)
서비스대상 서비스명
메시지이벤트 상세 내용
확인 여부읽음 상태 (✓ = 읽음, ● = 미읽음)

12.3 이벤트 유형 및 ITIL v4 우선순위​

v0.6.16부터 각 이벤트 유형에 ITIL v4 P1~P4 우선순위가 자동으로 할당됩니다.

유형배지ITIL 우선순위준거 표준의미
FAILURE빨간색P1 CriticalITIL v4 · ISO/IEC 20000-1:2018서비스 중단 — 즉각 대응 필수
RECOVERY_FAILED빨간색P1 CriticalITIL v4 · NIST SP 800-61 · ISO/IEC 27035자동 복구 실패 — 수동 개입 필수
CONFIG_UPDATE황금색P2 HighITIL v4 Change Enablement · ISO/IEC 27035설정 변경 — 승인 및 검토 필요
RECOVERY_SUCCESS초록색P3 MediumITIL v4 · NIST SP 800-61자동 복구 성공 — 모니터링 유지
SERVICE_RESTART청록색P4 LowITIL v4 Change Enablement · ISO/IEC 20000-1:2018서비스 재시작 — 확인·기록 수준
info

우선순위 체계는 ITIL v4 우선순위 매트릭스를 기반으로 설계되었습니다.

12.4 조치 권고 사항 다이얼로그​

알림 목록에서 행을 클릭하면 자동 조치 결과 리포트 다이얼로그가 열립니다. 이벤트 유형에 따라 ITIL v4 기반 맞춤형 조치 권고 사항이 자동으로 표시됩니다.

12.4.1 FAILURE — 장애 발생 리포트 (P1 Critical)​

서비스가 예기치 않게 중단된 경우입니다. 즉각적인 대응이 필요합니다.

표시 내용:

  • 우선순위 배지: P1 Critical (빨간색)
  • 준거 표준: ITIL v4 · ISO/IEC 20000-1:2018
  • 요약 메시지: "서비스가 예기치 않게 중단되었습니다. 즉각적인 조치가 필요합니다."

즉시 조치 타임라인:

시간조치
T+0인시던트 레코드 생성 및 담당자 즉시 알림
T+5분systemctl status <서비스> 로 실제 상태 확인
T+5분종속 서비스 및 영향 사용자 범위 파악
T+30분원인 미파악 시 에스컬레이션

에스컬레이션 기준: 30분 내 원인 미파악 시, 또는 24시간 내 2회 이상 발생 시

12.4.2 RECOVERY_FAILED — 복구 실패 리포트 (P1 Critical)​

자동 복구 시도가 실패한 경우입니다. 서비스가 중단 상태이며 수동 개입이 필수입니다.

표시 내용:

  • 우선순위 배지: P1 Critical (빨간색)
  • 준거 표준: ITIL v4 · NIST SP 800-61 Rev.3 · ISO/IEC 27035 Class 3-4
  • 요약 메시지: "자동 복구가 실패하였습니다. 서비스는 현재 중단 상태입니다. 즉시 수동 개입이 필요합니다."

즉시 조치 타임라인:

시간조치
T+0담당자 + 에스컬레이션 라인 즉시 통보 (P1)
T+0페일오버/우회 방안 즉시 활성화
T+0조치 전 완전한 진단 증거 수집 (NIST 800-61)
T+5분재시작 실패 원인 특정 (유닛파일/바이너리/디스크/포트)
danger

즉각 에스컬레이션 필수. 바이너리/설정 변조가 의심될 경우 보안팀에 즉시 알림 (ISO/IEC 27035).

12.4.3 CONFIG_UPDATE — 설정 변경 리포트 (P2 High)​

전문가 편집기를 통해 설정 파일이 수정된 경우입니다.

표시 내용:

  • 우선순위 배지: P2 — 승인 확인 필요 (황금색)
  • 준거 표준: ITIL v4 Change Enablement · ISO/IEC 27035
  • 요약 메시지: "설정 파일이 변경되었습니다. 변경 관리 프로세스에 따라 검토 및 승인을 확인하십시오."

즉시 조치:

  • CAB(Change Advisory Board) 승인 여부 확인
  • 변경 로그 및 롤백 계획 보관
  • 변경 영향 범위(종속 서비스) 재확인
warning

계획되지 않은 설정 변경이 감지되면 즉시 변경 원인을 파악하고 롤백을 검토하십시오.

12.4.4 RECOVERY_SUCCESS — 복구 성공 리포트 (P3 Medium)​

Watchdog 자동 복구가 성공한 경우입니다.

표시 내용:

  • 우선순위 배지: P3 모니터링 (초록색)
  • 준거 표준: ITIL v4 · NIST SP 800-61
  • 요약 메시지: "서비스가 자동 복구되었습니다. MTTR을 기록하고 재발 방지 조치를 검토하십시오."

즉시 조치:

  • MTTR(Mean Time To Recovery) 측정 및 기록
  • 인시던트 모니터링 상태 유지 (재발 감시)
  • PIR(사후 검토 회의) 일정 수립
tip

복구 성공 후에도 동일 서비스에서 24시간 내 재발 시 즉시 P1으로 에스컬레이션하십시오.

12.4.5 SERVICE_RESTART — 서비스 재시작 리포트 (P4 Low)​

tkctl 명령 등을 통해 서비스가 재시작된 경우입니다.

표시 내용:

  • 우선순위 배지: P4 — 정보 (청록색)
  • 준거 표준: ITIL v4 Change Enablement · ISO/IEC 20000-1:2018
  • 요약 메시지: "서비스가 재시작되었습니다. 계획된 재시작인지 확인하고 변경 레코드를 기록하십시오."

확인 사항:

  • 계획된 재시작인지 변경 레코드 확인
  • 서비스 헬스체크 및 종속 서비스 상태 확인
  • 재시작 사유, 담당자, 소요 시간 기록

12.4.6 알 수 없는 유형 — 기본 권고 사항 (P3 Medium)​

정의되지 않은 이벤트 유형에는 기본(폴백) 권고 사항이 표시됩니다.

표시 내용:

  • 우선순위 배지: P3 확인 필요 (회색)
  • 준거 표준: ITIL v4 Event Management
  • 요약 메시지: "시스템 이벤트가 기록되었습니다. 이벤트 내용을 확인하고 필요한 조치를 취하십시오."

12.5 필터링 기능​

유형별 필터​

상단 툴바의 드롭다운에서 이벤트 유형을 선택합니다.

옵션설명
모든 유형전체 보고 표시 (기본값)
장애 감지FAILURE 유형만 표시
복구 성공RECOVERY_SUCCESS 유형만 표시
복구 실패RECOVERY_FAILED 유형만 표시
설정 변경CONFIG_UPDATE 유형만 표시

검색​

검색 입력 필드에 서비스명 또는 메시지 내용을 입력합니다. 대소문자를 구분하지 않습니다.

12.6 읽음 관리​

개별 읽음 처리​

  1. 보고 테이블에서 행을 클릭합니다.
  2. 조치 권고 사항 다이얼로그가 표시됩니다.
  3. 모달이 열리면 해당 항목이 자동으로 읽음 처리됩니다.
  4. 닫기 버튼으로 모달을 닫습니다.

일괄 읽음 처리​

모두 읽음 버튼 클릭 시 모든 미확인 항목이 즉시 읽음 처리됩니다.

warning

감사 추적성 원칙: 복구 성공으로 배너가 사라지더라도, 운영자가 직접 읽음 처리하기 전까지는 미읽음 상태가 유지됩니다. 이는 사후 장애 이력 확인을 보장하는 설계입니다.

12.7 실시간 갱신​

항목값
폴링 주기10초
동작선택된 데이터 소스에서 최신 데이터 조회 및 갱신
종료 조건다른 메뉴로 이동 시 타이머 자동 해제

12.8 알림 데이터 영속성​

모든 이벤트는 SQLite audit_logs 테이블에 영구 기록됩니다.

필드설명
id고유 식별자
time이벤트 발생 시각 (RFC3339)
service대상 서비스명
type이벤트 유형
severityITIL v4 우선순위 (P1~P4)
message상세 메시지
is_healingWatchdog 자동 복구 중 여부
read읽음 여부

12.9 관련 API​

API메서드설명
/tkadmin/api/alertsGET메모리 캐시 최근 알림 조회 (최대 100건)
/tkadmin/api/audit-logsGETSQLite 감사 로그 전체 조회
/tkadmin/api/alerts/:idGET특정 알림 상세 조회
/tkadmin/api/alerts/readPOST모든 알림 일괄 읽음 처리
/tkadmin/api/alerts/unreadGET미읽음 건수 조회
